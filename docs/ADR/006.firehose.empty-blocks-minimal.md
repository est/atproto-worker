## Context

AT Protocol 的 `subscribeRepos` firehose 需要发送 `#commit` 事件，其中 `blocks` 字段应为 CAR 文件，包含 commit 对象和记录数据。

问题在于本项目的 CID 计算方式与 atproto 标准不同：

- 本项目：`CID = hash({op, collection, rkey, record, prev})`
- atproto 标准：`commit CID = hash({rev, prev, did, version, ops, ...})`

如果将当前格式的数据放入 CAR blocks，消费者验证 CID 时会失败（数据结构不匹配）。空 blocks 比错误 blocks 更安全。

## Decision

Firehose 的 `blocks` 字段使用**空 CAR 文件**（只有 header，无 blocks）。

已知能力边界：

- 可被 `unauthenticatedCommits: true` 的消费者接受
- 可获取记录内容和操作类型
- 无法验证 repo 完整性（需要 MST）
- 无法用于 repo 同步（需要完整 CAR）
- 不支持 `#sync`、`#identity`、`#account` 事件类型
- 不支持背压控制（ConsumerTooSlow）

## Consequences

优点：

- 不会向消费者发送无法验证的数据
- 实现简单，不引入复杂的 commit 对象构造
- 已在 README 中明确标注为已知限制

缺点：

- 无法被要求完整 CAR blocks 的消费者使用
- 无法用于 repo 同步
- 需要消费者设置 `unauthenticatedCommits: true`
