## Context

当前 journal 需要托管在外部静态文件服务（S3、GitHub Pages 等），用户需要：

1. 单独配置静态托管
2. 手动上传 journal.ndjson
3. 配置 `JOURNAL_URL` 指向托管地址
4. 手动触发 `/refresh` 或等待 cron 同步

这增加了部署复杂度，对非技术用户不友好。

## Decision (Draft)

使用 Cloudflare R2 存储 journal，消除外部静态托管需求：

- Worker 直接从 R2 读取 journal
- Web UI 直接写入 R2
- 移除 `JOURNAL_URL` 配置和 `/refresh` 端点（或改为内部使用）

配置变更：

```toml
[[r2_buckets]]
binding = "JOURNAL_BUCKET"
bucket_name = "atproto-journal"
```

## Consequences (Draft)

优点：

- 消除外部静态托管依赖
- 简化部署流程
- R2 与 Worker 同区域，延迟低

缺点：

- 与 Cloudflare R2 绑定
- 需要额外的 R2 配置
- 移除了"纯静态文件"的架构优势
