## Context

Worker 从 KV 缓存或 HTTP 源加载 journal 时，原始实现只做解析和索引，不做完整性校验。

由于 journal 来自不可信的静态托管源（S3、GitHub Pages 等），如果托管源、CDN 或缓存被污染，Worker 会把伪造事件作为真实 repo 向外暴露。

项目的核心卖点是"静态文件 + 不可信托管"，因此 journal 校验是安全底线。

## Decision

Worker 在 `load()` 和 `refresh()` 时都执行 journal 链式校验：

1. **prev 链校验**：`event[n].prev === event[n-1].cid`，确保事件顺序正确
2. **CID 校验**：对每个事件重新计算 CID，与事件中记录的 CID 比对，确保数据未被篡改

校验失败处理：

- `load()`：抛出错误，阻止加载污染数据
- `refresh()`：保留旧数据，避免服务中断

## Consequences

优点：

- 防止被污染的 journal 污染 Worker 服务
- 校验失败时有合理的降级策略
- CLI 和 Worker 使用相同的 CID 计算实现（`src/shared.js`）

缺点：

- 每次 load/refresh 都需要遍历所有事件计算 CID
- 大 journal 校验可能较慢
