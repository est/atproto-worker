## Context

AT Protocol firehose 的 `#commit` 事件对 `rev`、`since`、`blocks` 字段有格式要求：

- `rev`：应为 TID 格式（13 字符 base32-sortish，单调递增）
- `since`：应为前一个 `rev`（不是 CID）
- `blocks`：应为 CAR v1 文件格式

原始实现中：

- `rev` 使用数字字符串
- `since` 使用前一个事件的 CID
- `blocks` 为空字符串

## Decision

按 atproto 规范修正 firehose 字段格式：

- `rev`：使用 TID 格式（`generateTID()` 生成，保证单调递增）
- `since`：使用前一个事件的 `rev`（`event.prevRev`），首事件为 `null`
- `blocks`：使用 CAR v1 文件格式（`createCarFile()` 生成），目前为空 blocks（见 ADR-006）

CAR 文件结构：

- Header：`{version: 1, roots: [{$link: rootCid}]}`（CBOR 编码）
- Blocks：当前为空（event CID schema 与 atproto commit schema 不匹配）

同时，XRPC 端点 `listRepos` 和 `getLatestCommit` 的 `rev` 字段也改为 TID 格式。

## Consequences

优点：

- 符合 atproto firehose 协议规范
- 可被标准 atproto 消费者解析
- TID 保证时间有序，支持排序

缺点：

- blocks 为空，不支持 repo 完整性验证（见 ADR-006）
- CAR 文件生成增加了编码开销
