## Context

Worker 的 `/refresh` 端点需要从远程 HTTP 源重新拉取 journal，并识别新增事件。

原始实现通过比较"刷新前事件数量"和"刷新后事件数量"来计算新增事件：

```js
const newEvents = journal.events.slice(oldEventCount)
```

这隐含了一个强前提：journal 永远只追加、不重排、不重写。如果发布流程稍有偏离（如 journal 被重写），增量广播会漏事件或错事件。

## Decision

改为基于**最后一个事件的 CID** 来查找新增事件：

```js
const lastCidBefore = journal.events[journal.events.length - 1].cid
// ... refresh ...
const lastIdx = journal.events.findIndex(e => e.cid === lastCidBefore)
const newEvents = journal.events.slice(lastIdx + 1)
```

如果找不到 lastCid（journal 被完全重写），记录警告并广播所有事件。

## Consequences

优点：

- 可检测 journal 非 append-only 的情况
- 即使 journal 被重写也不会崩溃，只是记录警告
- 保持 firehose 广播的正确性

缺点：

- journal 被重写时会重复广播所有旧事件
- 仍建议发布流程保持 append-only
