## Context

Journal 是整个系统的唯一数据源。需要选择一种格式来存储事件序列。需求：

- Append-only（只追加，不修改）
- 可验证完整性（防篡改）
- 人类可读（便于调试）
- 支持增量拉取

候选格式：

- NDJSON（每行一个 JSON 对象）
- SQLite（D1）
- CBOR 二进制序列

## Decision

选择 NDJSON 格式，配合 CID 链式校验。

每个事件包含：

- `offset`：字节偏移量（用作 seq）
- `op`：操作类型（create/update/delete）
- `collection` / `rkey`：记录标识
- `record`：记录内容
- `cid`：基于 `{op, collection, rkey, record, prev}` 的 CID v1 (dag-cbor + sha-256)
- `prev`：前一个事件的 CID（形成链式结构）
- `sig`：对 CID 的 secp256k1 签名

链式校验逻辑：`event[n].prev === event[n-1].cid`。Worker 在 load 和 refresh 时都会验证整条链。

## Consequences

优点：

- 人类可读，便于调试
- Append-only 天然适合 git 管理
- CID 链提供完整性校验
- 字节偏移量作为 seq，确定性且单调递增

缺点：

- 不支持随机访问（需要顺序扫描）
- JSON 编码效率低于二进制格式
- 大 journal 需要全量加载到内存
