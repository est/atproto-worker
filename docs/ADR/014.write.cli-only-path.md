## Context

AT Protocol 的标准 PDS 通过 XRPC 写端点（`com.atproto.repo.createRecord`、`putRecord`、`deleteRecord`）接受写操作。

本项目的 Worker 是无状态的，不持有私钥（见 ADR-005）。如果支持 XRPC 写端点，需要额外的签名方案（如 Web UI 生成签名后提交），增加了复杂度和安全风险。

## Decision

CLI (`cli/seal.js`) 是**唯一的写入口**。Worker 的 XRPC 写端点统一返回 `501 MethodNotImplemented`。

写流程：

1. 用户在本地使用 CLI 创建记录（post、like、repost、follow）
2. CLI 签名并追加到 `journal.ndjson`
3. 用户上传 journal 到静态托管
4. Worker 通过 `/refresh` 或 cron 拉取新事件

CLI 支持的写操作：

- `post "text"` — 创建帖子
- `like <at-uri> <cid>` — 点赞（需要目标 CID）
- `repost <at-uri> <cid>` — 转发（需要目标 CID）
- `follow <did>` — 关注
- `rotate-key` — 轮换密钥

## Consequences

优点：

- Worker 安全模型简单（无私钥、无写操作）
- 所有写操作在本地完成，用户完全控制
- 不需要 Web 认证、CSRF 防护等

缺点：

- 所有写操作需要命令行
- 非技术用户无法使用（需要 Web UI，见 ADR-013）
- 发帖流程较慢（CLI → upload → refresh）
